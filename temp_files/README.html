# Knowledge Base Service
This service is used to store and retrieve knowledge base documents of ActionBoard.ai’s ActionBoard resources.

Codebase Guidelines

Please follow the following guidelines for the codebase.

1. Project Structure

The project is organized into the following folders:

  • .github: Contains the GitHub Actions workflows.
  • backend: Contains the backend code.
  • frontend: Contains the frontend code.
  • docs: Contains the project documentation.
  • README.md: Contains the project overview and codebase guidelines.

2. File Naming Conventions

  • Use kebab-case for file names.

3. Code Guidelines

  • Use TypeScript for the codebase.
  • Use ESLint for linting.
  • Use Prettier for formatting.
  • Use pascalCase for function and class names.
  • Use camelCase for variable names.
  • Use UPPERCASE for constants.
  • Use snake_case for database table names.

4. Branching Strategy

We use the following branching strategy:

  • main: The default branch.
  • deployment-dev: The development branch.
  • deployment-prod: The production branch.
  • deployment-stage: The staging branch.
  • feature/...: A feature branch.
  • bug/...: A bug fix branch.
  • hotfix/...: A hotfix branch.
  • release/...: A release branch.
  • docs/...: A documentation branch.
  • chore/...: A chore branch.

5. Commit Messages

We use the following commit message conventions:

  • feat: A new feature.
  • fix: A bug fix.
  • perf: A performance improvement.
  • refactor: A code change that neither fixes a bug nor adds a feature.
  • style: A code change that does not affect the meaning of the code (white-space, formatting, etc).
  • test: A code change that adds missing tests.
  • chore: A code change that neither fixes a bug nor adds a feature.
  • docs: A code change that updates the documentation.
  • build: A code change that affects the build system or external dependencies.
  • ci: A code change that affects the CI/CD configuration.
  • revert: A code change that reverts a previous commit.
  • workflow: A code change that affects the GitHub Actions workflows.

Example:

feat: add new feature #2
fix: fix bug #1
perf: improve performance #3
refactor: refactor code #4
style: style code #5
test: add tests #6
chore: chore code #7
docs: update documentation #8
build: build code #9
ci: update CI/CD configuration #10
revert: revert previous commit #11
workflow: update GitHub Actions workflows #12

At the trailing of the commit message, add the issue number.

6. Documentation

We use the following documentation conventions:

  • Use Markdown for the documentation.
  • Use the following format for the documentation:
  • # <title>
  • ## <subtitle>
  • ### <tertiary title>
  • #### <quaternary title>
  • ##### <quinary title>
  • ###### <senary title>
  • [description]
  • [details]
  • [example]
  • [see also]

7. Error Handling

We use the following error handling conventions:

  • Use the Error class to create custom errors.
  • Use the try-catch block to handle errors.
  • Use the throw statement to throw an error.
  • Use the console.error function to log an error.
  • Use the console.log function to log a message.
  • Use the console.warn function to log a warning.
  • Use the console.info function to log an info message.
  • Use the console.debug function to log a debug message.

8. Logging

We use the following logging conventions:

  • Use the console.log function to log a message.
  • Use the console.warn function to log a warning.
  • Use the console.info function to log an info message.
  • Use the console.debug function to log a debug message.

9. Environment Variables

We use the following environment variables:

  • NODE_ENV: The environment (e.g. development, production, staging, local).

Use .env.local for local development on your machine.
